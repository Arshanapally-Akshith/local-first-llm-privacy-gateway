# Dataset Card — Indian-PII Benchmark Dataset

**File:** `benchmarks/data/dataset.jsonl`
**Generator:** `benchmarks/generate/` (`python -m benchmarks.generate.build_dataset`)
**Status:** dataset generation only — no detection arm has been run against
this dataset yet (Presidio integration and the scorer are later Phase 5
tasks, not started).

This card documents the dataset as an artifact in its own right, independent
of any system that will later be measured against it. It exists because
BUILD.md requires one, and because a benchmark whose input data cannot be
explained and audited is not a benchmark a reviewer should trust — the same
standard `docs/DECISIONS.md` and `docs/LIMITATIONS.md` already hold the rest
of this project to.

## Purpose

A labeled evaluation set for benchmarking automated detection of Indian
personally identifiable information (PII) in free text — the input to
BUILD.md's Phase 5 four-arm ablation (stock Presidio / Presidio + custom
recognizers / Presidio + custom + GLiNER / this project's own cascade).
This card documents the *data*, not the arms or their results, neither of
which exist yet.

## Motivation

No public benchmark fairly measures Indian-specific structured PII (Aadhaar,
PAN, IFSC, UPI, vehicle registration) or Hinglish/Telugu-English
code-switched names against a properly configured baseline — the gap
`ARCHITECTURE.md`'s Executive Summary names as this project's actual
contribution ("the proxy is table stakes"). A benchmark requires labeled
data before it requires a detector; this dataset is that prerequisite.

## Generation methodology: slot-and-inject

BUILD.md is explicit about why: asking a model for sentences *and* their
entity spans in one step produces misaligned offsets on 10-20% of examples
in practice — a labeling problem that would cost a week of hand-correction
instead of building the system. This dataset instead separates the two
concerns entirely:

1. **Carrier sentences with `{ENTITY_TYPE}` slots are authored content**
   (`benchmarks/generate/templates.py`), written once, offline, in the role
   BUILD.md assigns to "the LLM" — by the assistant, during this build
   session — not fetched from a live API at generation time. This is a
   deliberate choice, not an oversight: this project requires no paid API
   key or network access to run, test, or benchmark (`CLAUDE.md` constraint
   #2), and a live generation call could return different phrasing on every
   run, which is incompatible with the byte-for-byte reproducibility this
   dataset guarantees. The template *set* was authored by an LLM in exactly
   the sense BUILD.md intends; it simply was not re-fetched at every build.
2. **Entities are injected programmatically** (`benchmarks/generate/entity_values.py`,
   `inject.py`) — every value is generated by code, using the gateway's own
   checksum functions, detector-documented structural rules, or candidate
   pools (see *Entity value generation* below), and substituted into a
   template's slots while a cursor tracks the exact character offset of
   each substitution as it happens.

The result: gold offsets require no annotation step and cannot silently
disagree with the text, because they are never computed by searching the
finished sentence for a value — only by recording where each value was
written as it was written. See *Exact-by-construction offsets* below.

### Entity value generation

Each of the 11 entity types is generated by reusing the gateway's own
production code, not a second implementation:

| Type | How the value is built | Reused from |
|---|---|---|
| AADHAAR | Verhoeff check digit computed over a payload fixed to UIDAI's reserved `9999...` prefix | `src/detect/tier1/checksum.py::verhoeff_generate_check_digit` |
| CARD | Luhn check digit over a 15-digit payload | `src/detect/tier1/checksum.py::luhn_generate_check_digit` |
| PAN | 5 letters (4th from the real Income Tax Dept. category-code set) + 4 digits + 1 letter | `src/detect/tier1/pan.py::PAN_CATEGORY_LETTERS` |
| IFSC | 4-letter bank code + reserved literal `0` + 6-char branch code | `IfscDetector`'s documented shape |
| UPI | Alphanumeric local part + dot-free PSP handle | `UpiDetector`'s documented shape; real NPCI PSP handle suffixes |
| VEHICLE_REG | State-code scheme: 2-letter state code + RTO code + series + number | `VehicleRegistrationDetector`'s documented shape |
| EMAIL | Local part + RFC 2606 reserved domain (`example.com`/`.net`/`.org`/`.edu`) | `EmailDetector`'s documented shape |
| PHONE | Leading digit 6-9 + 9 digits, optional `+91`/`91`/`0` prefix; regenerated on a coincidental cross-type Tier-1 checksum collision (see note below) | `PhoneDetector`'s documented shape |
| PERSON | Drawn from the gateway's own finite name-surrogate pool | `src/session/names.py` via `src/session/candidates.py::get_candidates()` |
| ORG | Drawn from the gateway's own finite org-surrogate pool (no real company names) | `src/session/org_names.py` |
| ADDRESS | Drawn from the gateway's own finite address-surrogate pool (generic street patterns, real city/state names) | `src/session/addresses.py` |

**Note on `PHONE` generation.** A `"91"`-prefixed phone number is exactly
Aadhaar-shaped (12 digits) and, roughly 1 in 10 times, also happens to be
Verhoeff- or Luhn-valid by pure chance — in which case the gateway's own
already-approved Tier-1 precedence tie-break (registration order) would
correctly, deterministically attribute that span to `AADHAAR`/`CARD`
instead of `PHONE`, since two different Tier-1 detectors both validate
the identical span. This was discovered by this benchmark's own real-arm
test suite (`docs/DECISIONS.md`, 2026-07-22, "Phase 5 Task 7" and its
follow-up entry), not assumed. `_generate_phone()` now regenerates (using
the same seeded `random.Random`, never reseeding it — determinism is
preserved) whenever a candidate would actually be reclassified by the
real cascade precedence rule, verified by literally running
`get_tier1_detectors()` + `precedence.resolve()` against each candidate
rather than a hand-derived approximation of that rule. Every gold `PHONE`
value in this dataset is therefore guaranteed, by construction, to
resolve as `PHONE` through the real cascade — not merely `PHONE`-shaped.

Every structured (Tier-1) value was validated, exhaustively across the whole
committed dataset, by running the actual production `Detector` for its type
against the full carrier sentence and confirming it returns exactly the
recorded gold span — not a second implementation of the detection rule, the
real one (`tests/unit/test_benchmark_build_dataset.py`).

## Supported entity types

All 11 types the gateway detects: `AADHAAR`, `PAN`, `IFSC`, `UPI`,
`VEHICLE_REG`, `CARD`, `EMAIL`, `PHONE`, `PERSON`, `ORG`, `ADDRESS`. Every
type appears in the dataset (165-825 gold occurrences per type — IFSC is
the least-represented at 165, PERSON the most at 825, driven directly by how
many templates each type happens to appear in).

## Language coverage

Two carrier-sentence varieties, tagged per example:

- **`en`** — plain English (32 templates, 1,760 examples).
- **`hi_en`** — Hindi/Telugu-English code-switched (20 templates, 1,100
  examples), **romanized (Latin script) throughout**. No Devanagari or
  Telugu script appears anywhere in this dataset — a deliberate scope limit
  for this first version, not a silent omission (see *Limitations*).

## Dataset size

2,860 examples = 52 templates × 55 instantiations per template (a fixed,
named constant — `benchmarks/generate/build_dataset.py::_INSTANTIATIONS_PER_TEMPLATE`
— chosen only to land inside BUILD.md's stated ~2-3k range for the current
template count). 4,235 total gold entity occurrences; most examples carry
one entity, a substantial minority carry two or three (multi-slot
templates), matching real request bodies where more than one sensitive
field often appears together.

## Exact-by-construction offsets — not annotated

Every gold span's `(start, end)` was recorded at the moment its value was
written into the growing output string, by tracking a cursor position —
never by searching the finished text for the value afterward. This matters
concretely: a naive `str.find()`-based labeler would be wrong the moment a
generated value (or a substring of it) also happens to appear elsewhere in
the same carrier sentence — for example the literal word "PAN" preceding a
`{PAN}` slot's own value. Slot-and-inject has no such failure mode, because
there is no separate labeling step that could disagree with the text — the
label *is* a direct record of what was written and where.

This is verified, not merely claimed: `test_every_gold_offset_is_exact_on_100_percent_of_examples`
asserts `text[start:end] == value` for every one of the 4,235 gold entities
in the committed dataset, exhaustively.

## Determinism and reproducibility

`DEFAULT_DATASET_SEED = 20260722` (`benchmarks/generate/seed.py`) is fixed
and arbitrary — chosen once, carrying no meaning beyond determinism, and
never to be changed casually (doing so regenerates a different — but
equally valid — dataset, invalidating comparability with any measurement
already taken against the current file). Every entity value is generated
from an `random.Random` instance seeded deterministically from this
constant and each example's position; nothing in the generator reads the
system clock, the network, or global `random` state.

Regenerate with:

```powershell
.\venv\Scripts\python.exe -m benchmarks.generate.build_dataset
```

This is proven byte-identical across independent runs — both by unit test
(`test_build_dataset_is_deterministic_across_independent_calls`,
`test_write_jsonl_is_byte_identical_across_independent_writes`) and, at
Task 2's completion, by an out-of-band SHA-256 comparison of two
independently-written files. Deleting `dataset.jsonl` and regenerating it
reproduces the identical committed bytes — the same "delete and regenerate,
get it back" discipline BUILD.md requires of the README's benchmark tables,
applied here to the dataset that will feed them.

## Synthetic data and ethical considerations

**All data in this dataset is synthetic. No real personal data of any kind
is included.**

- **AADHAAR**: every value is drawn from UIDAI's own documented, officially
  reserved `9999`-prefixed test-UID block (`docs/DECISIONS.md`, 2026-07-20;
  https://uidai.gov.in/en/916-developer-section/data-and-downloads-section/11350-testing-data-and-license-keys.html),
  not the general issuable space. This is a *stronger* guarantee than the
  gateway's own FF1 Aadhaar *surrogates* can offer for arbitrary real input
  (proven mathematically unable to guarantee reserved-range placement, same
  DECISIONS.md entry) — this dataset has no such constraint to satisfy,
  since every gold Aadhaar is chosen from the reserved block directly rather
  than derived from someone's real number.
- **CARD**: Luhn-valid numbers with no issuer-prefix correspondence to any
  real card network's actual allocation.
- **EMAIL**: every address uses an RFC 2606 reserved domain
  (`example.com`/`.net`/`.org`/`.edu`) — domains permanently reserved for
  documentation and guaranteed never to resolve to a real registrant.
- **PERSON / ORG / ADDRESS**: drawn from the gateway's own candidate pools,
  which themselves deliberately exclude real company names (no
  "Tata"/"Infosys"/etc.) and any single, uniquely identifying real street —
  the same reasoning `docs/DECISIONS.md` (Phase 4 Task 5) already applies to
  surrogate generation, reused here for gold values.
- **PAN / IFSC / UPI / VEHICLE_REG / PHONE**: structurally valid but not
  drawn from, or checked against, any real institution's actual issued
  records.

No human subject, real organization, or real individual is represented by
any value in this dataset. City and state names in `ADDRESS` values are
real (a fixed, public set — no different from a benchmark sentence naming a
real city), which is a disclosed, low-severity residual, not an oversight.

## Intended use

Benchmarking automated Indian-PII detection systems — specifically, the
four-arm ablation BUILD.md's Phase 5 defines (stock Presidio / Presidio +
custom recognizers / Presidio + custom + GLiNER / this project's cascade),
scored under the exact-span, exact-type criterion fixed in
`docs/DECISIONS.md` (2026-07-22) before any arm was implemented. Suitable
for any detector or pipeline that accepts free text and reports
`(start, end, entity_type)` spans.

## Out-of-scope use

- **Not a measurement of real-world deployment recall.** See *The
  optimistic-bound caveat* below — this is the single most important
  caveat this card makes.
- **Not a source of real PII** for any purpose, under any circumstance —
  there is none to extract.
- **Not an adversarial-robustness benchmark.** Every entity surface form is
  canonical (unobfuscated, correctly formatted). Spaced/dashed digits,
  number-words, transliteration, homoglyphs, and other bypass techniques
  are BUILD.md's Phase 6 adversarial suite's job, not this dataset's — using
  this dataset to claim adversarial robustness would be a category error.
- **Not a general-purpose NER training set.** The candidate pools behind
  `PERSON`/`ORG`/`ADDRESS` are small and finite (by design — they are the
  same pools the gateway uses for surrogate allocation); a model trained on
  this dataset would overfit to ~5,000 recycled name/org/address strings
  rather than learning to generalize.
- **Not evidence of code-switching diversity beyond what is authored here.**
  20 templates is a sample of Hinglish/Telugu-English phrasing patterns,
  not a corpus-scale account of code-switching variety.

## The optimistic-bound caveat

**Recall measured against this dataset is an optimistic bound on real-world
performance, not an estimate of it.** BUILD.md states this plainly and this
card repeats it deliberately, because it is the caveat most likely to be
skipped in a summary table:

- Carrier *phrasings* are diverse (52 templates spanning two languages and
  varied sentence structures), but entity *surface forms* are entirely
  ours — always canonical, always well-formed, always exactly the shape
  each Tier-1 detector's own regex expects. Real-world PII arrives spaced,
  dashed, mis-typed, OCR-mangled, transliterated, and embedded in far
  messier carrier text than any of the 52 templates here produce.
- A detector's recall on this dataset reflects its ability to find PII in
  its cleanest, most cooperative form. It says nothing about degraded
  recall against the messy forms Phase 6's adversarial suite exists
  specifically to measure.
- Any published number derived from this dataset must be reported and read
  as *"recall on canonical-form Indian PII in synthetic carrier text,"* not
  as *"recall on Indian PII."* Collapsing that distinction — in a table
  caption, a README line, or a conversation — overclaims what this dataset
  can support, the exact failure mode `CLAUDE.md`'s "modest about its own
  guarantees" standard exists to prevent.

## Limitations

- **Canonical-form entity values only.** No spaced/dashed digits (`1234
  5678 9012`), number-words, OCR-ish noise, or homoglyphs anywhere in this
  dataset — see *Out-of-scope use* and `docs/LIMITATIONS.md`'s existing
  "Detection is canonical-form only" entry, which this dataset's current
  scope matches exactly. Deliberately deferred at Task 2 (see that task's
  completion report), not yet scheduled to a specific later task.
- **Romanized code-switching only.** No Devanagari or Telugu script. Real
  Hinglish/Telugu-English text in the wild frequently mixes scripts; this
  dataset does not exercise that.
- **Closed, finite template set.** 52 templates is enough for BUILD.md's
  ~2-3k example target at 55 instantiations each, but represents a fixed,
  authored sample of carrier-sentence structure, not an open-ended one —
  expanding entity-type or phrasing coverage requires adding templates by
  hand, deliberately, not automatically.
- **PERSON/ORG/ADDRESS candidate pools are finite and reused from the
  gateway itself.** A detector could, in principle, be tuned (deliberately
  or accidentally, e.g. via memorization) against this specific finite pool
  in a way that would not generalize — a residual inherited from choosing
  to reuse the gateway's own surrogate pools for gold values (see
  *Generation methodology*), traded deliberately against authoring a wholly
  separate, unvalidated name list.
- **No arm has been run against this dataset yet.** This card describes the
  dataset only; no P/R/F1 number exists to report or misreport at this
  stage.
